### 问题 2：moveToTrash 缺少事务保护 — ✅ 已修复

- **代码位置**：[docs/1.sql 第 293-336 行（RPC 函数）](file:///d:/Codex01/01/docs/1.sql#L293) + [supabase-fileTree.ts 第 55-77 行](file:///d:/Codex01/01/repositories/supabase-fileTree.ts#L55)
- **现象描述**：旧流程为前端三次独立 HTTP 请求：① SELECT 查询 → ② INSERT trash → ③ DELETE file\_tree。如果②成功但③失败，trash 表留下孤儿记录。
- **根本原因**：Supabase 客户端无法跨多个 API 调用实现事务，三次操作非原子性执行。
- **修复方案（已实施）**：将三步骤合并为单个 PostgreSQL RPC 函数，在数据库层面保证原子性。

```sql
-- docs/1.sql — 新增 RPC 函数（第 13 节）
CREATE OR REPLACE FUNCTION public.move_to_trash(node_ids uuid[])
RETURNS void AS $$
DECLARE
  node record;
  schema_check boolean;
BEGIN
  -- 根节点校验
  SELECT EXISTS (
    SELECT 1 FROM public.file_tree
    WHERE id = ANY(node_ids) AND level = 1 AND parent_id IS NULL
  ) INTO schema_check;
  IF schema_check THEN
    RAISE EXCEPTION '不能删除根文件夹';
  END IF;

  FOR node IN
    SELECT ft.id, ft.user_id, ft.parent_id, ft.name, ft.type, ft.level,
           ft.created_at, ft.updated_at, d.content, d.word_count
    FROM public.file_tree ft
    LEFT JOIN public.documents d ON d.file_id = ft.id
    WHERE ft.id = ANY(node_ids)
  LOOP
    INSERT INTO public.trash (
      user_id, original_id, name, type, parent_id,
      original_level, content, word_count,
      original_created_at, original_updated_at
    ) VALUES (
      node.user_id, node.id, node.name, node.type, node.parent_id,
      node.level, node.content, node.word_count,
      node.created_at, node.updated_at
    );
  END LOOP;

  DELETE FROM public.file_tree WHERE id = ANY(node_ids);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

```typescript
// repositories/supabase-fileTree.ts — moveToTrash 方法
async moveToTrash(nodeIds: string[]): Promise<void> {
  const supabase = createClient()
  if (nodeIds.length === 0) return

  // 前端校验：禁止删除根节点
  const { data: roots } = await supabase
    .from('file_tree').select('id')
    .in('id', nodeIds).is('parent_id', null).eq('level', 1)
  if (roots && roots.length > 0) throw new Error('不能删除根文件夹')

  // RPC 调用：数据库原子化执行 SELECT → INSERT trash → DELETE CASCADE
  const { error: rpcErr } = await supabase.rpc('move_to_trash', { node_ids: nodeIds })
  if (rpcErr) throw rpcErr
}
```

**安全收益**：

1. 原子性：整个操作在单个 PostgreSQL 函数中执行，成功全成功，失败全回滚
2. 简化为 1 次 HTTP 调用（原为 3 次），减少网络延迟和故障面
3. RPC 函数内置根节点校验，与 RLS 策略形成双重防护

### 问题 3：useFolderItems / useBreadcrumb 不必要的重渲染 — ✅ 已修复

- **代码位置**：[hooks/use-file-tree.ts 第 1-67 行](file:///d:/Codex01/01/hooks/use-file-tree.ts#L1)
- **现象描述**：`useFolderItems`、`useBreadcrumb`、`useDocumentInfo`、`useFolderName` 四个 hook 每次渲染都创建新数组/对象引用，导致消费组件（`FolderView` 等）不必要的重渲染。

```typescript
export function useFolderItems(): FolderItem[] {
  const tree = useFileTreeStore((s) => s.getTree())
  const selectedNode = useFileTreeStore((s) => s.getSelectedNode())

  return useMemo(() => {
    const node = selectedNode ?? tree[0]
    if (!node || node.type !== "folder" || !node.children) return []
    return node.children.map((child) => ({
      id: child.id,
      name: child.name,
      type: child.type as "folder" | "file",

      createdAt: formatTimestamp(child.createdAt),

}

    return buildBreadcrumb(tree, selectedNodeId)
  }, [tree, selectedNodeId])
}
```

### 问题 4：文档保存无并发控制 — ✅ 已修复

```typescript
// repositories/supabase-fileTree.ts — upsertDocument
async upsertDocument(fileId: string, content: DocNode, wordCount: number, expectedUpdatedAt?: string): Promise<void> {
  const supabase = createClient()

  if (expectedUpdatedAt) {
    const { data: existing } = await supabase
      .from('documents')
      .select('updated_at')
      .eq('file_id', fileId)
      .single()

    if (existing && existing.updated_at !== expectedUpdatedAt) {
      throw new Error('CONFLICT: 文档已被其他会话修改')
    }
  }

  const { error } = await supabase
    .from('documents')
    .upsert({ file_id: fileId, content, word_count: wordCount }, { onConflict: 'file_id' })
  if (error) throw error
}
```

```typescript
// store/editorStore.ts — performSave
async function performSave() {
  const { content, title, isDirty, loadedUpdatedAt } = get()
  if (!isDirty) return

  // ...
  try {
    await saveDocument({
      nodeId, title, content, wordCount: words,
      expectedUpdatedAt: loadedUpdatedAt ?? undefined,  // ← 携带版本号
    })
    const now = new Date().toISOString()
    set({ isSaved: true, isDirty: false, wordCount: words, loadedUpdatedAt: now })
  } catch (err) {
    if ((err as Error).message.startsWith('CONFLICT')) {
      set({ isSaved: false, isDirty: true, error: '文档已被其他会话修改，请刷新后重试' })
    } else {
      set({ isSaved: false })
    }
  }
}
```

**机制说明**：

1. `loadEditorFromTree` 加载文档时记录 `node.updatedAt` → `loadedUpdatedAt`
2. 每次自动保存/手动保存时携带 `expectedUpdatedAt`
3. Repository 层先 SELECT `updated_at`，与 `expectedUpdatedAt` 比对
4. 不匹配 → 抛出 `CONFLICT` 错误 → UI 提示用户刷新
5. 匹配 → 正常 upsert，保存成功后更新本地 `loadedUpdatedAt`

**注意**：`updated_at` 在 file\_tree 和 documents 两个表中各有一份，且触发器 `handle_new_file` 可能不同步。**后续建议**：将乐观锁改为检查 `documents.updated_at` 或使用 `xmin` 系统列。

### 问题 5：cancelCreate 可能误删已持久化的根节点 — ✅ 已修复

- **代码位置**：[fileTreeStore.ts 第 228-248 行](file:///d:/Codex01/01/store/fileTreeStore.ts#L228)
- **现象描述**：用户首次使用（`DEFAULT_ROOT_ID` 场景），创建第一个文件时触发 `addChildToTree` 将 `DEFAULT_ROOT_ID` 替换为真实 UUID。此时点击取消新建，`cancelCreate` 调用 `deleteNodeService(_tree, nodeId)`，但此时父节点可能已经是真实 UUID 而非 `DEFAULT_ROOT_ID`。如果 nodeId 恰好指向根节点，则整个树被删除。
- **根本原因**：
  1. `cancelCreate` 无 root 保护，直接调用 `deleteNodeService`
  2. `deleteNodeService` 的根保护条件是 `tree.length === 1 && tree[0].id === nodeId`，但此时树长度 > 1（已创建了子节点），该条件失效
- **修复方案（已实施）**：在 `cancelCreate` 中增加两层根节点保护。

```typescript
// fileTreeStore.ts — 修复后
cancelCreate: (nodeId: string) => {
  // 第一层：DEFAULT_ROOT_ID 硬编码检查
  if (nodeId === DEFAULT_ROOT_ID) {
    set({ creatingNodeId: null })
    return
  }

  const { _tree, selectedNodeId } = get()
  // 第二层：isRootNode 语义检查（level=1 且无 parent_id）
  const node = findNode(_tree, nodeId)
  if (node && isRootNode(node)) {
    set({ creatingNodeId: null })
    return
  }

  const result = deleteNodeService(_tree, nodeId)
  if ("error" in result) return
  set({
    _tree: result.newTree,
    creatingNodeId: null,
    ...(selectedNodeId === nodeId
      ? { selectedNodeId: _tree[0]?.id ?? null, currentView: "folder" as const }
      : {}),
  })
},
```

**防护层级**：
| 层 | 检查 | 触发场景 |
| - | --- | --- |
| 1 | `nodeId === DEFAULT_ROOT_ID` | 首次用户，虚拟根节点 |
| 2 | `isRootNode(node)` | 真实 UUID 根节点 |
| 3 | `deleteNodeService` 内部 `tree.length === 1 && tree[0].id === nodeId` | 剩余树的唯一节点 |

### 问题 7：moveToTrash 未处理子节点 — ✅ 已修复

- **代码位置**：[docs/1.sql 第 293-355 行](file:///d:/Codex01/01/docs/1.sql#L293) + [fileTreeStore.ts 第 216 行](file:///d:/Codex01/01/store/fileTreeStore.ts#L216)
- **现象描述**：旧方案前端通过 `collectDescendantIds` 收集所有子孙 ID 传给 RPC，依赖前端本地树状态的正确性。如果前端树状态与数据库不同步，会有遗漏。
- **修复方案（已实施）**：RPC 函数内置递归 CTE 收集子孙节点，前端只需传入根节点 ID。

```sql
-- RPC 内置递归 CTE
WITH RECURSIVE descendants AS (
  SELECT id FROM public.file_tree WHERE id = ANY(node_ids)
  UNION ALL
  SELECT ft.id
  FROM public.file_tree ft
  INNER JOIN descendants d ON ft.parent_id = d.id
)
SELECT array_agg(id) INTO all_ids FROM descendants;

-- 将所有节点（根 + 所有子孙）一次性归档到 trash
-- CASCADE DELETE 仅需删除根节点，子节点由 FK ON DELETE CASCADE 自动清理
```

```typescript
// deleteNode 简化为只传根节点 ID
await fileTreeRepo.moveToTrash([nodeId])
// RPC 内部递归 CTE 自动收集所有子孙节点，前端只需传根节点 ID
```

**收益**：
1. 前端不再依赖本地树状态收集子孙 ID，以数据库为唯一真相源
2. 消除 `collectDescendantIds` 的 import（不再需要）
3. RPC 内部 SELECT + INSERT + DELETE 仍在同一事务中

## 七、性能分析

| 操作                     |   复杂度   | 瓶颈                                                                |
| ---------------------- | :-----: | ----------------------------------------------------------------- |
| `buildTree`            |   O(n)  | 2 次遍历，均摊 O(n)，无瓶颈                                                 |
| `findNode`             | O(n) 最坏 | 递归全树遍历，树深 < 5 层 + 节点数 < 1000 时可接受                                 |
| `buildBreadcrumb`      | O(n) 最坏 | 同上，但在 hook 中每次 `selectedNodeId` 变化时重新计算                           |
| `collectDescendantIds` |   O(n)  | 同上                                                                |
| `useFolderItems`       |   O(m)  | m = 当前文件夹子节点数，但每次渲染都创建新数组                                         |
| `TreeItem.memo`        |    ✅    | 正确使用 `memo`，但 `expandedIds`（Set）作为 selector 会导致当前层所有 TreeItem 重渲染 |

**优化建议**：

1. `useBreadcrumb` 和 `useFolderItems` 加 `useMemo`（见问题 3）
2. 可以考虑对 `expandedIds` 使用 `useShallow` 比较，或将其转换为原始值数组（`string[]`）以便 Zustand 的默认浅比较生效
3. 树节点规模 < 1000 个时，当前性能可接受。超过 2000 个节点后需要考虑虚拟滚动
