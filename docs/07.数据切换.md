## 1. 架构概览

项目采用 **Repository 模式**统一管理所有数据访问。无论 UI 层、状态层还是业务逻辑层，**都不直接访问 `data/` 目录**，而是通过 `repositories/` 提供的接口操作数据。

```
┌──────────────────────────────────────────────┐
│  UI 层            pages, components           │  ← 只展示，不读数据
└────────────────────┬─────────────────────────┘
                     │
┌────────────────────┴─────────────────────────┐
│  状态层           store/                      │  ← import { xxxRepo } from "@/repositories"
└────────────────────┬─────────────────────────┘
                     │
┌────────────────────┴─────────────────────────┐
│  业务逻辑层       lib/services/               │  ← import { xxxRepo } from "@/repositories"
└────────────────────┬─────────────────────────┘
                     │
┌────────────────────┴─────────────────────────┐
│  数据访问层       repositories/              │  ← 唯一的 data/ 调用者
│  ┌─────────────┬──────────────┬───────────┐  │
│  │ userRepo    │ fileTreeRepo │ trashRepo │  │  ← 模块级单例
│  │ workspace   │              │           │  │
│  └──────┬──────┴──────┬───────┴─────┬─────┘  │
└─────────┼─────────────┼─────────────┼────────┘
          │             │             │
┌─────────┴─────────────┴─────────────┴────────┐
│  数据存储         data/                       │
│  mockUsers │ mockFileTree │ mockTrash │ ...  │
└──────────────────────────────────────────────┘
```

**核心原则**：`data/` 只被 `repositories/` 中的 4 个 Mock 实现文件调用。其余所有文件通过 `repositories/index.ts` 导出实例访问。

## 2. 代码导航：快速找到你要改的代码

### 场景 → 文件 速查表

| 你想做的事 | 去这个文件 | 改什么 |
|-----------|-----------|--------|
| 新增一种数据实体 | `repositories/types.ts` | 新增接口 |
| 新增数据实体 Mock 实现 | `repositories/mock-xxx.ts`（新建） | 实现接口方法 |
| 注册新 Repository 实例 | `repositories/index.ts` | 新增 `export const` |
| 修改用户数据格式 | `types/auth.ts` | 修改 `User` 接口 |

| 从 Mock 切到真实 API | `repositories/index.ts` | 替换 `new MockXxx()` 为 `new ApiXxx()` |
| 页面需要新数据 | 对应 `page.tsx` | 从 `@/repositories` import |
| Store 需要新数据 | 对应 `store/xxxStore.ts` | 从 `@/repositories` import |
| Service 需要新数据 | 对应 `lib/services/xxx.ts` | 从 `@/repositories` import |

## 3. 目录结构

```
repositories/
  types.ts              ← 接口定义（IUserRepository / IFileTreeRepository / ...）
  mock-users.ts         ← MockUserRepository 实现
  mock-fileTree.ts      ← MockFileTreeRepository 实现
  mock-trash.ts         ← MockTrashRepository 实现
  mock-workspace.ts     ← MockWorkspaceRepository 实现
  index.ts              ← 创建单例 + 统一导出

data/
  mockUsers.ts          ← 原始数据 + 低层存储函数（被 Mock Repository 调用）
  mockFileTree.ts       ← 原始数据 + 低层存储函数
  mockTrash.ts          ← 原始数据 + 低层存储函数
  workspace-mock.ts     ← workspace 常量数据
```

**关键约定**：

- `data/` 文件**只导入类型**和内部工具，**不导入任何其他项目模块**
- `repositories/mock-*.ts` **只导入** `@/repositories/types` + `@/data/*`
- 其余消费者**只导入** `@/repositories`

## 4. 四个 Repository 接口详解

接口定义文件：[repositories/types.ts](file:///d:/Codex01/01/repositories/types.ts)

### 4.1 IUserRepository

管理用户数据的增删改查和本地持久化。

```ts
interface IUserRepository {
  MOCK_USERS: User[]                                   // 所有注册用户的数组
  findUserByEmail(email: string): User | undefined      // 按邮箱查找用户
  addUser(user: User): void                             // 添加新用户到列表
  getUserFromStorage(): User | null                     // 从 localStorage 读取当前登录用户
  updateUserInStorage(user: User): void                 // 更新 localStorage 中的用户信息
}
```

**被哪些文件使用**：
- `lib/services/auth-service.ts` — 登录验证、注册
- `lib/services/userService.ts` — 更新用户信息、持久化

### 4.2 IFileTreeRepository

管理文件树的增删改。

```ts
interface IFileTreeRepository {
  getTree(): TreeNode[]              // 获取完整文件树
  setTree(newTree: TreeNode[]): void // 覆盖整个文件树
  resetTree(): void                  // 重置为初始 Mock 数据
}
```

**被哪些文件使用**：
- `store/fileTreeStore.ts` — 所有文件树操作（创建、重命名、删除、恢复等）

### 4.3 ITrashRepository

管理垃圾箱数据。

```ts
interface ITrashRepository {
  getTrash(): TrashItemData[]              // 获取所有垃圾项
  setTrash(items: TrashItemData[]): void   // 覆盖垃圾箱数据
  addToTrash(item: TrashItemData): void    // 添加项目到垃圾箱
  removeFromTrash(id: string): void        // 按 ID 移出垃圾箱
  clearTrash(): void                       // 清空垃圾箱
}
```

**被哪些文件使用**：
- `store/trashStore.ts` — 垃圾箱状态管理
- `store/fileTreeStore.ts` — 删除节点移入垃圾箱（`trashRepo.addToTrash`）
- `lib/services/trashService.ts` — 恢复、清空、彻底删除

### 4.4 IWorkspaceRepository

提供工作区的常量 Mock 数据。

```ts
interface IWorkspaceRepository {
  mockUser: UserInfo                       // 当前 Mock 用户信息
  mockMenuActions: DropdownMenuItem[]      // 编辑器菜单操作项
}
```

**被哪些文件使用**：
- `app/(main)/workspace/page.tsx` — 传入 `SidebarLayout` 和 `DocumentEditor`
- `app/(main)/trash/page.tsx` — 传入 `SidebarLayout`

## 5. 如何新增数据访问

假设你需要新增一个「标签」功能，需要管理标签数据。

### 步骤 1：定义类型（如果还没有）

在 `types/` 下创建或扩展类型文件：

```ts
// types/tag.ts
export interface Tag {
  id: string
  name: string
  color: string
}
```

### 步骤 2：在 `repositories/types.ts` 新增接口

```ts
export interface ITagRepository {
  getAll(): Tag[]
  add(tag: Tag): void
  remove(id: string): void
}
```

### 步骤 3：创建 Mock 实现

新建 `repositories/mock-tags.ts`：

```ts
// repositories/mock-tags.ts
import type { Tag } from "@/types/tag"
import type { ITagRepository } from "@/repositories/types"

const memoryTags: Tag[] = [
  { id: "tag-1", name: "重要", color: "#EF4444" },
  { id: "tag-2", name: "待办", color: "#F59E0B" },
]

export class MockTagRepository implements ITagRepository {
  getAll(): Tag[] {
    return memoryTags
  }
  add(tag: Tag): void {
    memoryTags.push(tag)
  }
  remove(id: string): void {
    // 注意：这里不要用 filter 重新赋值给 const
    // 直接 splice 删除
    const idx = memoryTags.findIndex(t => t.id === id)
    if (idx !== -1) memoryTags.splice(idx, 1)
  }
}
```

### 步骤 4：在 `repositories/index.ts` 注册实例

```ts
import { MockTagRepository } from "@/repositories/mock-tags"

export const tagRepo: ITagRepository = new MockTagRepository()
```

### 步骤 5：在 Store / Service 中使用

```ts
import { tagRepo } from "@/repositories"

// 在任何需要的地方调用
const tags = tagRepo.getAll()
tagRepo.add({ id: "tag-3", name: "归档", color: "#6B7280" })
```

## 6. 如何修改已有数据访问

### 场景 A：修改 Mock 初始数据

直接编辑 `data/mockUsers.ts`、`data/mockFileTree.ts` 等。

> ⚠️ 初始数据中的对象可能已被 `structuredClone` 复制到内存变量，**修改初始数组后需刷新页面**才能看到效果。

### 场景 B：扩展接口方法

1. 在 `repositories/types.ts` 接口中新增方法签名
2. 在对应的 `repositories/mock-xxx.ts` 实现类中添加方法体
3. 在消费者（Store/Service）中调用新方法

### 场景 C：修改数据结构

1. 在 `types/` 中修改 TypeScript 接口（如 `User` 加字段）
2. 在 `data/` 中更新 Mock 初始数据（补充新字段值）
3. 在 `repositories/mock-xxx.ts` 中更新相关方法（如果新字段影响了存取逻辑）

## 7. 如何从 Mock 切换到真实 API

这是 Repository 模式的**核心价值**——切换数据源只需修改 `repositories/index.ts` 一个文件。

### 7.1 切换前

当前 `repositories/index.ts` 关键部分：

```ts
// repositories/index.ts
import { MockUserRepository } from "@/repositories/mock-users"
import { MockFileTreeRepository } from "@/repositories/mock-fileTree"

export const userRepo: IUserRepository = new MockUserRepository()
export const fileTreeRepo: IFileTreeRepository = new MockFileTreeRepository()
// ...
```

### 7.2 创建 API 实现

以 User 为例，新建 `repositories/api-users.ts`：

```ts
// repositories/api-users.ts
import type { User } from "@/types/auth"
import type { IUserRepository } from "@/repositories/types"

export class ApiUserRepository implements IUserRepository {
  // 注意：API 模式下列表数据从服务端获取，不维护 MOCK_USERS
  MOCK_USERS: User[] = []

  async findUserByEmail(email: string): Promise<User | undefined> {
    const res = await fetch(`/api/users?email=${encodeURIComponent(email)}`)
    if (!res.ok) return undefined
    return res.json()
  }

  async addUser(user: User): Promise<void> {
    await fetch("/api/users", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(user),
    })
  }

  getUserFromStorage(): User | null {
    // API 模式可用 token 从服务端拿用户
    // 或从 localStorage 读 token，再请求 /api/me
    return null // 由 AuthContext 初始请求处理
  }

  updateUserInStorage(user: User): void {
    // API 模式：发起 PUT /api/users/:id 请求
    fetch(`/api/users/${user.id}`, {
      method: "PUT",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(user),
    })
  }
}
```

### 7.3 切换

修改 `repositories/index.ts`：

```ts
// repositories/index.ts

// Mock → API：只需替换这 2 行 import
// import { MockUserRepository } from "@/repositories/mock-users"
import { ApiUserRepository } from "@/repositories/api-users"

// 其余保持不变...
export const userRepo: IUserRepository = new ApiUserRepository()
```

### 7.4 注意事项

| 问题 | 说明 |
|------|------|
| **异步方法** | Mock 方法是同步的，如果真实 API 是异步的，需要同时改接口和调用方 |
| **类型一致性** | 接口返回类型不变，调用方无需修改 |
| **逐步切换** | 可以逐个 Repository 切换，无需一次性全部切换 |
| **测试** | Mock 实现可保留用于单元测试 |

## 8. 数据流示意图

### 用户登录流程

```
LoginForm (UI)
  → useAuth().login(email, password)
    → auth-service.validateLogin(email, password)
      → userRepo.MOCK_USERS.find(...)
        → data/mockUsers.MOCK_USERS
    → auth-service.createSession(user)
      → localStorage.setItem(...)
    → setUser(user)  ← AuthContext 更新状态
```

### 文件树操作流程

```
FileTree / FolderListItem (UI) 点击重命名
  → folder-list-item.tsx: handleRename()
    → window.prompt() 获取新名称
    → useFileTreeStore().commitRename(id, name)
      → renameNodeService(tree, nodeId, newName)  ← fileTreeService 纯函数
        → 返回新的 TreeNode[]
      → fileTreeRepo.setTree(result.newTree)
        → data/mockFileTree.setTree(...)
    → set({ tree: result.newTree })
```

### 删除文件到垃圾箱流程

```
TreeItemMenu (UI) 点击删除
  → useFileTreeStore().deleteNode(nodeId)
    → moveToTrash(tree, nodeId)  ← fileTreeService 纯函数
      → 从树中移除节点，生成 TrashItemData
    → trashRepo.addToTrash(trashItem)
      → data/mockTrash.addToTrash(...)
    → fileTreeRepo.setTree(newTree)
      → data/mockFileTree.setTree(...)
```

## 9. 常见问题

### Q: 为什么 data/ 下的函数不能在其他地方直接调用？

违反单向依赖原则。数据层是最底层，应该被 Repository 层封装。如果 Store 直接调 `data/mockFileTree.getTree()`，切换真实 API 时需要修改 Store 代码，失去了 Repository 抽象的意义。

### Q: 接口方法全是同步的，真实 API 是异步的怎么办？

接口可以改成返回 `Promise`。当前 Mock 实现返回同步值（TypeScript 允许 `Promise<T>` 兼容 `T`），未来 API 实现返回 `Promise`，调用方使用 `await` 即可。

### Q: 可以只切换部分 Repository 吗？

可以。`repositories/index.ts` 支持混合导出：

```ts
export const userRepo = new ApiUserRepository()      // 已切换到 API
export const fileTreeRepo = new MockFileTreeRepository() // 仍用 Mock
```

### Q: 如何处理 localStorage？

当前 Mock 实现中，`userRepo.getUserFromStorage()` 和 `updateUserInStorage()` 操作 localStorage。切换到 API 后，这两方法改走服务端 API，不再读写 localStorage（可用于存储 token）。

### Q: 新增 Repository 后忘记注册怎么办？

TypeScript 会在编译时报错，因为调用方 import 时会找不到导出变量。`pnpm lint` 也能捕获。
